# Blog Interactive Interview

This is an interview skill, not an implementation skill.

Interview the user relentlessly before implementation. Ask questions one at a time and provide your recommended answer for each question.

Do not edit files, create components, replace markers, or start implementation unless the user explicitly says to skip the interview or explicitly approves the final agreed direction after the interview.

If a question can be answered by reading the post or codebase, inspect it instead of asking.

If the user invocation sounds like "blog interactive postprocess on <post>", that is not implementation approval. Start the interview.

## Marker Syntax

```md
[Custom: <mini-description-of-the-interactive-widget>]
[R: Reason as to why it would make sense to add here]
[V: Text description of the visual representation of the widget].
```

Find the post in `src/content/blog/*`. If the user did not name a post, ask which post to use.

Find the first `[Custom:` marker by default. Parse its following `[R: ...]` and `[V: ...]` lines as draft hints, not requirements. They were likely generated by an LLM and may be wrong, weak, redundant, or not worth implementing.

Challenge the premise before design:

- Should this interactive element exist at all?
- Does `R` justify adding interaction here?
- Is `V` the right visual, or is a different visual/no visual better?
- What should the reader understand after using it?
- What can the reader click, drag, filter, compare, reveal, or change?
- What exact claims/data from the post may the component encode?
- What makes the component worth keeping?

## After Approval

Only after explicit approval:

- Build one component for the one marker being processed.
- Do not create one catch-all component for multiple markers.
- Put it at `src/components/blog/<blog-slug>/<interactive-component-name>.astro`.
- Follow root `AGENTS.md`.
- Consult current Astro docs before using framework-specific APIs or patterns.
- Use custom elements for client-oriented behavior.
- Write TypeScript inside `<script>` tags.
- Prefer Tailwind classes. Use Tailwind `@apply` only when needed. Custom CSS is last resort.
- Replace the exact marker block in place with the component usage.
- Do not move components to the top, bottom, or a gallery section unless explicitly asked.
- Preserve surrounding post text, paragraph flow, and headings.
- Run relevant checks when feasible.
